# Repo Structure

This repo centers on a ROS 2 + Webots simulation for multi-robot mapping and trust-based map-poisoning defense.

## Top Level

- `README.md` gives the entry points and the current project summary.
- `docs/` holds the short project notes, structure guide, Webots guide, verification guide, and the recurring-bugs note in [docs/lessons_learned.md](lessons_learned.md).
- `scripts/` holds the launch, verification, and bridge helper scripts.
- `src/` holds the ROS packages.
- `webots/` holds the Webots worlds and controller code.
- `docker/` holds the Docker image and compose files.
- `build/`, `install/`, and `log/` are generated by `colcon` and can be removed safely.

## ROS Packages

### `src/robot_patrol_msgs`

- `msg/MapUpdate.msg` defines the shared map-update message used by the defense work.
- `CMakeLists.txt` and `package.xml` define the message package.

### `src/robot_patrol_node`

This is the main ROS 2 package for the project.

Important subfolders:

- `launch/` contains the launch entry points.
- `config/` contains RViz configs, the multi-robot JSON config, and Nav2 parameters.
- `robot_patrol_node/` contains the executable Python nodes.
- `resource/` contains the package marker.
- `test/` contains the lint and style tests.

Key launch files:

- `amcl_stack.launch.py` starts the AMCL smoke-test stack.
- `mapping_stack.launch.py` starts the single-robot map-building path.
- `multi_robot_mapping.launch.py` starts the two-robot shared-mapping demo, the static `/map` server, the per-robot bridge nodes, the per-robot map builders, the shared-map merge nodes, the confidence overlays, and the fake-obstacle injectors.
- `fake_obstacle_injector.launch.py` is the standalone injector entry point.
- `rviz.launch.py` opens RViz with the requested config.
- `nav2_stack.launch.py` and `nav2_with_amcl.launch.py` keep the AMCL/Nav2 path available.

Key runtime nodes:

- `udp_bridge_node.py` bridges Webots packets to ROS topics.
- `map_builder_node.py` builds each robot's local occupancy grid and confidence grid.
- `map_merge_node.py` merges the per-robot maps, applies trust weighting, and handles fake obstacle reports.
- `confidence_marker_node.py` renders the trust-weighted overlay in robot-specific colors.
- `fake_obstacle_injector_node.py` publishes click-driven or manual fake obstacle claims.
- `checkpoint_patrol_node.py`, `initial_pose_publisher_node.py`, `localized_pose2d_node.py`, `navigation_diagnostics_node.py`, and `pose_to_odom_node.py` support the patrol and AMCL flows.

Important configs:

- `config/multi_robot_config.json` is the current two-robot baseline.
- `config/multi_robot_robot_1_view.rviz` and `config/multi_robot_robot_2_view.rviz` are the per-robot shared-map RViz layouts.
- `config/multi_robot_base.rviz` is the shared base layout.
- `config/multi_robot_shared_map.rviz` is the older shared-map view.
- `config/amcl.rviz`, `config/default.rviz`, and `config/office_amcl.rviz` support the single-robot smoke tests.

## Webots

### Worlds

- `webots/worlds/testRvizMap/` is the AMCL smoke-test world.
- `webots/worlds/testBuildingMapForRobot/` is the live map-building test world.
- `webots/worlds/TestCombineRvizMap/` is the two-robot shared-mapping sandbox.
- `webots/worlds/TestFakeObstacle/` is the fake-obstacle shared-mapping sandbox.
- `webots/worlds/office/`, `webots/worlds/confusingMaze/`, and `webots/worlds/sandbox/` are the other demo worlds.

### Controllers

- `webots/robot_controllers/patrol_robot/patrol_robot.py` is the Nav2-capable patrol controller.
- `webots/robot_controllers/user_controlled_robot/user_controlled_robot.py` is the operator-driven controller.
- `webots/worlds/controllers/*/*.py` are thin wrappers that let Webots find the real controller modules.

### World-local assets

Each world can keep its own generated AMCL map under `webots/worlds/<world>/amcl_map/`.

## Scripts

- `quick_test.sh` is the main smoke test.
- `runTestFakeObstacle.sh` starts the two-robot shared-mapping fake-obstacle demo.
- `runTestCombineRvizMap.sh` starts the combined two-robot RViz demo.
- `runOffice.sh`, `runConfusingMaze.sh`, `runSandbox.sh`, and `runTestBuildingMapForRobot.sh` start the other world-specific demos.
- `verify.sh` runs the headless checks.
- `start_ros2_stack.sh` starts the ROS 2 stack inside Docker.
- `check_ros2_stack.sh`, `watch_topic.sh`, `echo_topic_once.sh`, and `send_test_bridge_packet.py` are debugging helpers.

## Data Flow

1. Webots runs a world and controller.
2. The controller sends pose and scan data to the ROS bridge in Docker.
3. The bridge republishes `/robot_pose`, `/scan`, and `/odom`.
4. The mapping nodes build per-robot live maps and confidence maps.
5. The merge nodes create per-robot shared views.
6. RViz displays the shared live maps, confidence overlays, and static `/map` layer.
7. The fake-obstacle injector can publish `/map_updates` for shared-map experiments.

## Current Baseline

The current two-robot shared-mapping baseline is driven by `src/robot_patrol_node/config/multi_robot_config.json`.

- `robot_1` starts at `x=2.0`, `y=2.0`, `yaw=0.0` and uses `wasd`.
- `robot_2` starts at `x=-2.0`, `y=-2.0`, `yaw=1.5708` and uses the arrow keys.
- Robot 1 currently trusts Robot 2 at `0.80`.
- Robot 2 currently trusts Robot 1 at `0.20`.
- The shared-map demo keeps the static `/map` server active so the gray floor remains visible behind the live overlays.
