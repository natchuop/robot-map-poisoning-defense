# Set Up A Webots World For ROS 2, Docker, And RViz

This document gives an AI enough context to add or repair a Webots world in this repository.

## How The Pieces Connect

Webots runs on the laptop and simulates the robot, sensors, and world geometry. A Webots Python controller reads GPS, IMU, and LiDAR data, converts that data into a small JSON packet, and sends it over TCP to Docker on port `5005`.

Docker runs ROS 2 Jazzy. The default quick test starts an AMCL localization stack: the bridge listens on port `5005`, publishes `/robot_pose` and `/scan`, a helper mirrors Webots pose into `/odom`, `map_server` publishes the known world map, AMCL consumes `/scan`, and RViz loads `amcl.rviz` with fixed frame `map`.

Data flow:

```text
Webots world -> Webots Python controller -> tcp://*:5005 -> Docker ROS 2 bridge -> /robot_pose + /scan -> pose_to_odom + AMCL/map_server -> /map + /odom + /tf -> RViz
```

The older live mapping mode is still available with `RMPD_TEST_MODE=mapping bash scripts/quick_test.sh`; that mode uses `map_builder_node.py` to build `/map` from Webots pose and LiDAR.

## Webots World Requirements

The `.wbt` file must include a robot with a controller name that matches a controller folder.

Example:

```text
TurtleBot3Burger {
  translation 0 0 0
  rotation 0 1 0 0
  controller "patrol_robot"
  extensionSlot [
    GPS {
    }
    InertialUnit {
    }
    RobotisLds01 {
    }
  ]
}
```

The current controller expects these Webots device names:

```text
gps
inertial unit
LDS-01
LDS-01_main_motor
LDS-01_secondary_motor
left wheel motor
right wheel motor
```

If the robot or sensor PROTO uses different device names, update `webots/controllers/<controller_name>/<controller_name>.py` to match the new names.

## Controller Folder Rules

Keep real controller code in the top-level Webots controller folder:

```text
webots/controllers/<controller_name>/<controller_name>.py
```

Because this repo keeps worlds under `webots/worlds/<world_name>/`, Webots also needs a small runnable wrapper under:

```text
webots/worlds/controllers/<controller_name>/<controller_name>.py
```

That wrapper should only forward to the real controller. Do not duplicate robot behavior there, or maps can drift onto different controller versions.

If creating a new controller, create both the real controller and the wrapper, then set the world robot's `controller` field to that folder name.

## Bridge Packet Contract

The Webots controller sends newline-delimited JSON over TCP by default.

Default controller settings:

```text
WEBOTS_BRIDGE_PROTOCOL=tcp
WEBOTS_BRIDGE_TARGETS=172.28.64.1,127.0.0.1
WEBOTS_BRIDGE_PORT=5005
```

The bridge packet must include:

```json
{
  "pose": {
    "x": 0.0,
    "y": 0.0,
    "theta": 0.0
  },
  "scan": {
    "angle_min": -3.14159,
    "angle_max": 3.14159,
    "angle_increment": 0.01745,
    "range_min": 0.05,
    "range_max": 4.0,
    "scan_time": 0.064,
    "ranges": [1.0, 1.1, 1.2]
  }
}
```

ROS publishes this as:

```text
/robot_pose  geometry_msgs/Pose2D
/scan        sensor_msgs/LaserScan
/map         nav_msgs/OccupancyGrid
/odom        nav_msgs/Odometry
/tf          odom -> base_link
/tf_static   map -> odom, base_link -> laser
```

## Docker Requirements

The main compose file must expose port `5005` for TCP and UDP:

```yaml
ports:
  - "5005:5005/udp"
  - "5005:5005/tcp"
```

Windows RViz GUI uses:

```text
docker-compose.wslg.yml
```

macOS should not use `docker-compose.wslg.yml`. macOS RViz GUI from Docker may need XQuartz or another X11 setup.

## ROS 2 Package Requirements

If adding new launch or config files, include them in:

```text
src/robot_patrol_node/setup.py
```

The current package installs:

```python
('share/' + package_name + '/launch', ['launch/amcl_stack.launch.py']),
('share/' + package_name + '/launch', ['launch/mapping_stack.launch.py']),
('share/' + package_name + '/launch', ['launch/rviz.launch.py']),
('share/' + package_name + '/config', ['config/amcl.rviz']),
('share/' + package_name + '/config', ['config/default.rviz']),
```

The default quick test uses:

```bash
bash scripts/quick_test.sh
```

The container-side helper builds and then launches either AMCL mode or mapping mode:

```bash
bash scripts/start_ros2_stack.sh
```

## RViz Requirements

`amcl.rviz` should keep:

```text
Fixed Frame: map
Map topic: /map
LaserScan topic: /scan
AMCL particles topic: /particle_cloud
TF enabled
```

If RViz opens but the view is empty, first verify that `/robot_pose`, `/scan`, `/map`, `/odom`, `/tf`, and `/tf_static` are publishing before changing RViz settings.

## How To Add A New World

1. Create a new folder:

```text
webots/worlds/<world_name>/
```

2. Put the `.wbt` file in that folder.

3. Add or reuse a controller:

```text
webots/controllers/<controller_name>/<controller_name>.py
webots/worlds/controllers/<controller_name>/<controller_name>.py
```

For most maps, reuse `webots/controllers/patrol_robot/patrol_robot.py`.

4. In the `.wbt` robot node, set:

```text
controller "<controller_name>"
```

5. Make sure the robot includes GPS, IMU, and LiDAR devices, or update the controller to match the new sensors.

6. Add or update a known AMCL map for the world.

For the current test world, this lives under:

```text
webots/worlds/testRvizMap/amcl_map/
```

If creating a new world, keep the AMCL map beside that world, for example:

```text
webots/worlds/<world_name>/amcl_map/
```

7. Run the quick test:

```bash
bash scripts/quick_test.sh
```

The script starts Docker, launches the ROS 2 AMCL stack, opens Webots, waits for bridge packets and TF, and opens RViz.

8. Check topics if needed:

```bash
docker exec -it ros2_dev bash
source /opt/ros/jazzy/setup.bash
source "${RMPD_CONTAINER_WORKSPACE:-/workspace}/install/setup.bash"
ros2 topic echo /robot_pose
ros2 topic echo /scan
ros2 topic echo /map
ros2 topic echo /odom
ros2 run tf2_ros tf2_echo map base_link
```

## Verification

Before committing world or controller changes, run:

```bash
bash scripts/verify.sh
```

Expected result:

```text
Results: ... passed, 0 failed
All checks passed.
```

This proves Docker builds, ROS 2 builds, the bridge accepts packets, `/robot_pose` and `/scan` publish, `/map` publishes, Nav2/AMCL packages are installed, and `rviz2` is installed.
